# ClippysRevenge — Hobbyist LLM Training Pipeline

A complete, from-scratch pipeline for training a small language model on conversational English, biased toward programming and Rust development. Built to run on consumer hardware — no cloud GPUs required.

> **Status:** Phase 1 data pipeline complete. Tokenizer, model architecture, pretraining loop, SFT loop, and inference REPL are all implemented. Actively scaling toward the 50M parameter target.

---

## The Idea

Most "train your own LLM" projects either hand-wave the data collection or assume you have a server rack. This project takes the opposite approach: a careful, resumable, deduplicated data pipeline that runs in the background on a laptop while you do other things.

The end goal is a **~50M parameter model** that can discuss and write Rust code at a senior developer level. Getting there requires two phases:

### Phase 1 — General Language Foundation

Train on broad conversational English so the model understands how humans communicate: asking questions, explaining concepts, debugging problems, casual chat. This is the base layer — without it, fine-tuning on code produces a model that can generate Rust but can't explain it or respond to natural language.

Data sources:
- **Synthetic conversations** generated by local models via LM Studio (`generate_convos.py`)
- **Synthetic lectures** — single-turn in-depth monologues on Rust topics (`generate_lecture.py`)
- **Real Rust source code** scraped from crates.io, GitHub, official docs, and docs.rs (`scraper.py`)
- **Official documentation** — The Rust Book, Reference, Nomicon, RFCs, and more
- **External datasets** — curated third-party Rust datasets (see [External Datasets](#external-datasets))

### Phase 2 — Rust Specialization *(active)*

Fine-tune the Phase 1 model on curated, validated Rust examples. The backbone is `rust_categories.jsonl` — 1,000 granular categories covering every major Rust pattern, concept, and ecosystem crate. Local and frontier models generate examples against these categories (`generate_examples.py`), every one of which is then run through a validation pipeline to ensure it compiles clean. This is where the model learns to *write* Rust, not just talk about it.

## Requirements

### Data generation

Python 3.10+. No GPU needed — all inference runs inside LM Studio via its HTTP API.

```bash
pip install requests rich
```

You also need **LM Studio** with the local server running. Any instruction-tuned model works. The scripts have been tested with Tesslate Tessa-Rust-T1 7B, Qwen3, Nemotron, and Dolphin variants. Non-thinking models are strongly recommended for bulk generation — thinking models burn tokens on chain-of-thought that gets stripped and never makes it into training data.

> **Note:** You do not need to pre-load a model before running the generators. They manage loading and unloading themselves via the LM Studio API. You only need the LM Studio server process running.

### Model pipeline

```bash
pip install torch numpy tqdm tokenizers transformers beautifulsoup4 lxml
```

ROCm (AMD) or CUDA. The training loop is written for ROCm on RDNA3 (gfx1103) but runs on CUDA without changes. CPU training is not practical at any useful scale.

## Quick Start

### 1. Start LM Studio server

**Desktop app:** Go to the Developer tab and toggle the server on (default port 1234).

**Headless / CLI:** If you have the `lms` CLI installed:

```bash
lms daemon up
lms server start
```

### 2. Find your model IDs

```bash
curl http://localhost:1234/api/v0/models
```

Look for the `"id"` field — that's the exact string to pass to `--models`.

### 3. Generate conversations

```bash
# Single model
python generate_convos.py generate --models "your-model-id" --count 50000 --rotate-every 200 --output-dir ./output

# Multiple models — rotates between them for variety
python generate_convos.py generate --models "model-id-1" "model-id-2" --count 50000 --rotate-every 150 --output-dir ./output
```

Resume an interrupted run by running the exact same command — it counts what's already on disk and continues from there.

### 4. Generate lectures

Lectures are single-turn monologues: one authoritative, in-depth explanation per record. Good complement to multi-turn conversations for pretraining signal.

```bash
python generate_lecture.py generate --models "your-model-id" --count 5000 --rotate-every 50 --output-dir ./lectures
python generate_lecture.py stats  --output-dir ./lectures
python generate_lecture.py sample --output-dir ./lectures
```

### 5. Check your progress

```bash
python generate_convos.py stats --output-dir ./output
python generate_convos.py stats --output-dir ./output --verbose    # per-category detail

# Spot-check random samples
python generate_convos.py sample --output-dir ./output --n 3
python generate_convos.py sample --output-dir ./output --category rust_ownership --n 2
```

### 6. Scrape real Rust data

```bash
# Top 500 crates by all-time downloads
python scraper.py crates --top 500 --output-dir ./rust_data

# rust-lang org + ecosystem repos (GitHub token strongly recommended)
python scraper.py github --token ghp_yourtoken --output-dir ./rust_data

# Official docs: The Book, Reference, Nomicon, Compiler Dev Guide, and more
python scraper.py docs --output-dir ./rust_data

# Rendered API docs from docs.rs for 70+ top crates
python scraper.py docsrs --output-dir ./rust_data

# Run all four in sequence
python scraper.py all --token ghp_yourtoken --output-dir ./rust_data

# Check what you've collected
python scraper.py stats --output-dir ./rust_data
```

## Model Pipeline (End-to-End)

Once data is collected, the full pipeline from raw text to a trained model runs in order:

### Step 1 — Train the tokenizer

```bash
python train_tokenizer.py --data_dir C:/llm/data --output_dir C:/llm/tokenizer
```

Trains a BPE tokenizer on the full corpus. Default vocab size is 40,960 with special tokens pre-baked (`<|user|>`, `<|assistant|>`, `<|code|>`, `<|endturn|>`, etc.) so you never need to retrain if you add new data.

Optional flags: `--vocab_size` (default: 40960), `--min_frequency` (default: 5), `--min_chars` (default: 50).

### Step 2 — Prepare training data

```bash
python prepare_data.py \
    --data_dir C:/llm/data \
    --tok_dir  C:/llm/tokenizer \
    --output_dir C:/llm/data/prepared
```

Tokenizes all JSONL records, shuffles at document level, splits into train/val, and writes packed `uint16` memory-mapped binaries (`train.bin`, `val.bin`, `meta.json`). Documents are concatenated with `<|eos|>` separators — no padding. The training loop slices this into `(seq_len + 1)`-token windows.

Optional flags: `--seq_len` (default: 2048), `--val_fraction` (default: 0.005), `--seed` (default: 42), `--encode_batch` (default: 512).

### Step 3 — Pretrain

```bash
# Fresh run
python train.py --data_dir C:/llm/data/prepared --out_dir C:/llm/runs/run1

# Resume
python train.py --data_dir C:/llm/data/prepared --out_dir C:/llm/runs/run1 --resume
```

BF16 mixed precision, gradient checkpointing, gradient accumulation, AdamW with cosine LR + linear warmup, gradient clipping. Saves `ckpt_best.pt` whenever validation loss improves and `ckpt_latest.pt` after every save interval.

Default training config (tuned for ~15M model on iGPU):

| Setting | Default | Notes |
|---|---|---|
| Micro batch | 2 | Sequences per GPU step |
| Grad accum | 64 | Effective batch = 128 sequences |
| Max steps | 12,500 | ≈3.3B tokens at 128 seq × 2048 tok |
| Peak LR | 3e-4 | Cosine decay to 3e-5 |
| Warmup | 200 steps | Linear ramp |
| Grad clip | 1.0 | |
| Weight decay | 0.1 | |

### Step 4 — (Phase 2) Generate and validate Rust examples

```bash
# Generate broken→fixed Rust examples with a local model
python generate_examples.py --model "your-model-id" --count 3

# Validate raw datasets — only pure-std Rust (no external crates)
python validate_dataset.py --input-dir datasets/raw --output-dir datasets/validated --workers 4

# Validate frontier datasets — resolves external crate dependencies via cargo add
python validate_frontier.py --input-dir datasets/frontier --output-dir datasets/rust --workers 2
```

The validation pipeline runs `cargo fmt`, `cargo clippy -D warnings`, `cargo build` (MSRV), and `cargo test` where tests are present. Only examples that pass all checks are kept. `validate_frontier.py` additionally resolves labeled crate dependencies automatically before building.

### Step 5 — (Phase 2) Prepare SFT data

```bash
python prepare_sft_data.py \
    --data_dir   C:/llm/data/sft \
    --tok_dir    C:/llm/tokenizer \
    --output_dir C:/llm/data/sft_prepared
```

Formats structured `broken_code → fixed_code` records using the special token conversation template. Writes parallel token and mask binaries — the mask ensures loss is computed only on assistant-turn tokens, not on the prompt. Records longer than `seq_len` are truncated from the prompt side, preserving the full assistant response.

### Step 6 — Fine-tune

```bash
# Masked instruction SFT (preferred — use prepare_sft_data.py output)
python sft.py \
    --data_dir  C:/llm/data/sft_prepared \
    --base_ckpt C:/llm/runs/run1/ckpt_best.pt \
    --out_dir   C:/llm/runs/sft1

# Continued pretraining SFT (raw corpus — lower LR than pretraining)
python sft.py \
    --data_dir  C:/llm/data/prepared \
    --base_ckpt C:/llm/runs/run1/ckpt_best.pt \
    --out_dir   C:/llm/runs/sft1 --resume
```

Loads a pretrained checkpoint and fine-tunes with a reset optimizer at a lower LR (peak 3e-5 vs 3e-4 for pretraining). Automatically detects masked vs. raw data from directory contents.

### Step 7 — Inference

```bash
# Interactive REPL
python generate.py --checkpoint C:/llm/runs/run1/ckpt_best.pt

# Run automated Rust test suite
python generate.py --checkpoint C:/llm/runs/run1/ckpt_best.pt --test

# Run one category only
python generate.py --checkpoint ckpt_best.pt --test --category ownership

# Save a benchmark report
python generate.py --checkpoint ckpt_best.pt --bench
```

Since this is a raw pretrained model (no SFT yet), it completes text rather than following instructions — give it a Rust prefix and it continues it. The REPL supports commands for adjusting temperature, top-k, repetition penalty, and more (type `:help` inside the REPL).

## Model Architecture

`model.py` implements a decoder-only transformer. The current `ModelConfig` defaults to a 15M parameter configuration used for development and iteration; the project target is **~50M parameters**.

**Current config (15M — development):**

| Hyperparameter | Value |
|---|---|
| Layers | 6 |
| d_model | 256 |
| Attention heads | 4 (head_dim = 64) |
| FFN hidden (SwiGLU) | 704 |
| Vocab size | 40,960 |
| Max seq len | 2,048 |
| Parameters | ~15.4M |
| BF16 size | ~29 MB |

**Architecture details:**
- Pre-norm with RMSNorm
- SwiGLU feed-forward (hidden ≈ 8/3 × d_model)
- RoPE positional embeddings (θ = 10,000)
- Tied input/output embeddings (lm_head shares weights with embed)
- `F.scaled_dot_product_attention` — no flash-attn dependency (ROCm-safe)
- Gradient checkpointing via `torch.utils.checkpoint`

**ROCm / gfx1103 notes:** No `torch.compile()` (patchy Triton support on RDNA3). No fused AdamW kernel (crashes on gfx1103 ROCm nightlies). BF16 is natively supported. Do not set `expandable_segments` — not supported on this platform.

Scaling to ~50M is a matter of increasing `d_model`, `n_heads`, `n_layers`, and `ffn_hidden` in `ModelConfig`. No architectural changes required.

## Tokenizer

A custom BPE tokenizer trained on the full Rust corpus. Key properties:

- **Vocab size:** 40,960 (chosen as a multiple of 64 for GPU alignment)
- **Special tokens** (baked in at positions 0–10 — never need to retrain):

| Token | ID | Purpose |
|---|---|---|
| `<\|pad\|>` | 0 | Padding |
| `<\|bos\|>` | 1 | Beginning of sequence |
| `<\|eos\|>` | 2 | End of sequence / document separator |
| `<\|unk\|>` | 3 | Unknown token fallback |
| `<\|system\|>` | 4 | System prompt start |
| `<\|user\|>` | 5 | User turn start |
| `<\|assistant\|>` | 6 | Assistant turn start |
| `<\|endturn\|>` | 7 | End of any turn |
| `<\|code\|>` | 8 | Code block start |
| `<\|endcode\|>` | 9 | Code block end |

Handles all four corpus schema variants automatically: conversations (turns array), lectures/docs (source: docs/crates/github), snippets (Strandset format), and scraped HTML (stripped via BeautifulSoup).

## How the Generator Works

### Weighted Categories

`categories.jsonl` contains 100 conversation categories, each with a `weight` field. Higher weight = more conversations generated from that category. The distribution is intentionally skewed toward Rust and programming:

| Subcategory | Categories | Approx. weight share |
|---|---|---|
| Rust | 32 | 47% |
| Software Engineering | 12 | 10% |
| Systems Programming | 7 | 7% |
| Tooling | 6 | 5% |
| Computer Science | 4 | 4% |
| Career / Educational / etc. | 39 | 27% |

The general conversation categories (casual chat, explaining concepts, career discussions) give the model broad language understanding — a model trained *only* on Rust code can generate Rust but can't hold a conversation about it.

### Prompt Diversification

Every generation call builds a uniquely-framed prompt by independently randomizing:

- **User persona** — beginner, Python dev learning Rust, senior C++ engineer, bootcamp grad, etc.
- **Scenario** — debugging production under pressure, following a tutorial, contributing to open source, etc.
- **Conversation angle** — specific aspect of the topic (e.g. for `rust_ownership`: "move semantics", "two mutable references", "Clone vs Copy", etc.)
- **Opening style** — pasting an error message, asking a wrong question, comparing to something they know, etc.
- **Style modifiers** — include code snippet, user pushes back, assistant uses an analogy, etc.

This creates tens of millions of unique prompt framings per category before any repetition becomes probable.

### Deduplication

Every saved conversation is fingerprinted as the SHA-256 of its normalized first user turn. Fingerprints are persisted to `.hashes/{category}.txt` inside your output directory. On every generation:

1. If the fingerprint already exists → discard silently and regenerate with a fresh prompt
2. The dedup store is loaded at startup so resume runs inherit full history

This ensures you never get the same conversation twice even across multiple sessions.

### Parse Repair

Small models frequently return malformed JSON. Rather than discarding these, the parser runs a repair pipeline — in order, stopping at first success:

1. Strip markdown fences and extract the JSON object
2. Attempt straight `json.loads`
3. Repair single quotes (Python dict syntax instead of JSON)
4. Repair truncated output — hit `max_tokens` mid-stream; tries 7 suffix closings
5. Repair unescaped inner quotes inside string values
6. Repair fused turns — recover both turns from objects with duplicate role/content keys
7. Normalize wrong role names (`Human`, `AI`, `Expert` → `user`/`assistant`)
8. Merge consecutive same-role turns
9. Drop trailing user turns (odd turn count)
10. Drop leading assistant turns
11. Unwrap nested structures

Any parse that still fails after all repair attempts is logged in full to `parse_failures.log` so you can review and add new repairs as needed.

### Model Rotation

Pass multiple model IDs to `--models` and set `--rotate-every N`. The script will unload the current model, load the next, generate N conversations, and repeat round-robin. This produces variety in tone, style, and phrasing. It also means your laptop's RAM gets freed between models since only one is in memory at a time.

### Thinking Model Support

Qwen3, DeepSeek-R1, and other reasoning models are fully supported. The script handles both output patterns:

- `delta.reasoning_content` — LM Studio's dedicated reasoning field
- `<think>...</think>` tags in `delta.content` — including tags that span multiple streaming chunks

Thinking tokens are shown live in the terminal but **stripped from saved training data** — your JSONL files will never contain `<think>` artifacts.

> **Recommendation:** Use non-thinking models for bulk generation. Thinking tokens cost generation time and are always stripped — they produce zero training signal.

### Schema

Each entry contains everything needed to drive a code generation prompt:

```json

  "prompt_focus": "Writing code that relies on NLL to release borrows before end of scope; patterns that didn't compile in pre-NLL Rust.",
  "difficulty": "intermediate",
  "tags": ["lifetimes", "NLL", "borrow-checker"],

```

| Field | Description |
|---|---|
| `prompt_focus` | Specific aspects and patterns to cover — the direct input to the generation prompt |
| `difficulty` | `beginner` / `intermediate` / `advanced` — 18% / 45% / 38% distribution |
| `target_entries` | Minimum examples per category (5 baseline; scale up for high-weight categories) |

| `turns_min/max` | Conversation turn count range for dialogue-format examples |

### Scale

At 100 examples per category: **100,000 total examples** — roughly 30–50M tokens of verified Rust code. Combined with the conversational and lecture data and the scraped documentation, this forms the core of a training set where the model sees real, validated Rust across every major pattern and crate in the ecosystem.

### Validation Pipeline

All examples generated from this category list are run through `validate_frontier.py` before being added to the training set:

```
cargo fmt --check
cargo clippy -- -D warnings
cargo build  (MSRV)
cargo test   (if tests present)
```

`validate_frontier.py` automatically resolves external crate dependencies using `cargo add` before building, so examples that use tokio, serde, axum, and 70+ other common crates validate correctly. `validate_dataset.py` is the stricter variant: only pure-std Rust (no external crates) passes.

## Scraper Details

`scraper.py` collects from four sources:

### crates.io

Fetches crate lists from `crates.io/api/v1/crates` (rate limited to 1 request/sec per crawlers policy), then downloads tarballs from `static.crates.io` (CDN, no rate limit). Collects top-N by downloads, category-based listings, and recently-updated crates. Extracts `.rs` source files, `Cargo.toml`, `build.rs`, and READMEs. Filters out generated code, tiny files, and files with no real Rust constructs. Prefers latest stable version over pre-releases.

### GitHub

Downloads full repo archives for 15+ `rust-lang` org repos (the compiler, Cargo, Clippy, rust-analyzer, The Book, Reference, Nomicon, RFCs, and more) and 100+ curated ecosystem crates across async, web, CLI, databases, crypto, embedded, game-dev, data processing, and more.

Get a free GitHub personal access token at [github.com/settings/tokens](https://github.com/settings/tokens) — no scopes needed, public repos only. Without a token you're rate-limited to 60 requests/hour. With one: 5,000/hour.

### Docs

Scrapes chapter-by-chapter from 25+ official and community Rust books and guides with code-block-preserving HTML extraction:
- The Rust Book, Reference, Rustonomicon, Async Book, Edition Guide
- Compiler Dev Guide, API Guidelines, Atomics & Locks, and more

### docs.rs

Renders API documentation for 70+ top crates directly from docs.rs. Captures type signatures, trait implementations, module docs, and inline examples — things that don't appear in source code but are exactly what a model needs to understand how an API is meant to be used.

All four scrapers support **resume** via a manifest file in your output directory. Kill at any time, restart with the same command — already-completed items are skipped instantly.

## Hardware Reality Check

Built and tested on a Ryzen 7 laptop with integrated Radeon RDNA3 graphics (780M iGPU) and 16GB unified RAM.

**Generation speed** depends entirely on your model and hardware:
- 4B model on integrated graphics: ~5–15 tok/s → ~2–4 minutes per conversation
- 7B model on a mid-range dedicated GPU: ~20–40 tok/s → ~30–60 seconds per conversation
- 13B model on a high-end GPU: ~15–25 tok/s → ~45–90 seconds per conversation

**Training a 50M parameter model:**
- Model weights (BF16): ~96 MB
- Optimizer state (AdamW, FP32 moments): ~384 MB additional
- With gradient checkpointing: activations are recomputed rather than stored — peak VRAM stays manageable on an iGPU
- At Chinchilla-optimal ratios, a 50M model wants ~1B training tokens; the current corpus is on track for this

**Target dataset size:** For a 50M parameter model, aim for ~50K–100K synthetic conversations plus the scraped and external data for Phase 1. That's well within reach on a laptop running generation in the background.

At 3 minutes per conversation on slow hardware: 50K conversations ≈ ~100 hours of compute. Run it in the background over a couple of weeks.

**Scraping** is I/O bound, not compute bound — runs fine on any hardware.

## Why Not Just Fine-Tune an Existing Model?

You could fine-tune Llama or Mistral on Rust data and get a capable Rust assistant in a weekend. That's the practical path if your goal is a working tool.

This project has a different goal: understanding the full stack from raw text to a working language model. The synthetic data pipeline, the tokenizer, the architecture, the training loop — building each piece teaches you something that downloading a checkpoint doesn't. The Rust specialization is the destination, but the journey is the point.
